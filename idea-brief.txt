# PRODUCT

Build **ProofYield**, a testnet-only, non-custodial DeFi treasury research and transaction-planning agent.

ProofYield reads a wallet across three EVM testnets, discovers configured yield opportunities, rejects unsafe or ineligible choices using deterministic policy code, explains the selected and rejected candidates, simulates an action, prepares an unsigned transaction for the user's browser wallet, verifies the submitted transaction, and generates a verifiable Decision Receipt.

This is not a profit-guarantee bot. It optimizes transparent, risk-adjusted testnet decisions. The LLM coordinates and explains; deterministic TypeScript code validates, scores, authorizes, and builds transaction requests.

# NON-NEGOTIABLE ARCHITECTURE

- Build exactly **one NitroStack MCP server/application**.
- Use the generated Starter project's installed NitroStack version and conventions.
- Use TypeScript, NitroStack modules/tools/resources/prompts/widgets, and Zod validation.
- Use one server with internal modules; do not create multiple MCP deployments.
- No database, OAuth, backend signer, private key, seed phrase, mainnet, Solana, Jupiter, bridge, autonomous trading loop, or unrestricted calldata.
- The browser wallet will sign and submit transactions. The MCP server only reads, reasons deterministically, simulates, prepares allowlisted unsigned transactions, and verifies results.
- Never fabricate contract addresses, balances, APYs, block numbers, simulation results, or transaction hashes.
- Missing protocol configuration must produce a structured `NOT_CONFIGURED` or `ADAPTER_UNAVAILABLE` result, not fake data.
- Do not make network calls during module import or production build.

# MCP TOOLS

Implement these tools with precise descriptions, Zod input validation, examples, and structured outputs.

## 1. `system_getStatus`

Read-only, no input. Return:

- application name and version;
- one-server architecture;
- phase/status;
- supported chains and protocols;
- whether each RPC, USDC address, Aave adapter, and demo vault is configured;
- tool names;
- schema version;
- dynamic ISO timestamp.

Never return secret values or RPC URLs.

## 2. `portfolio_getSnapshot`

Input:

```json
{
  "walletAddress": "0x...",
  "chainIds": [11155111, 84532, 80002]
}
```

`chainIds` is optional and defaults to all three.

For each chain independently, read live:

- chain ID/name;
- current block number;
- native balance;
- test-USDC balance when configured;
- observed timestamp;
- provenance.

One failed or unconfigured chain must return a per-chain error without destroying successful results from other chains.

## 3. `opportunities_scan`

Input:

```json
{
  "walletAddress": "0x...",
  "assetId": "USDC",
  "amount": "1000000",
  "allowedChainIds": [11155111, 84532, 80002]
}
```

Use an adapter interface. Query configured adapters live. Return normalized candidates with provenance, observed rates, liquidity/availability where obtainable, and adapter status. Do not claim that testnet APY is economically meaningful; label it as an observed testnet value. Include `DO_NOTHING` even when every adapter is unavailable.

## 4. `strategy_createPlan`

Input:

- wallet snapshot or its hash/reference;
- opportunities or their hash/reference;
- user policy;
- amount to allocate.

Default policy:

- allowed chains: all three;
- allowed protocols: `aave-v3`, `demo-erc4626`, `do-nothing`;
- asset: USDC;
- keep at least 20% liquid;
- maximum 60% in one protocol;
- require manual approval;
- reject stale data older than 120 seconds;
- require a meaningful improvement threshold.

Apply hard eligibility gates before ranking. Stable rejection codes must include:

```text
CHAIN_NOT_ALLOWED
PROTOCOL_NOT_ALLOWED
ASSET_NOT_ALLOWED
CONTRACT_NOT_ALLOWLISTED
DATA_STALE
PROVENANCE_MISSING
RISK_LIMIT_EXCEEDED
LIQUIDITY_INSUFFICIENT
CONCENTRATION_LIMIT_EXCEEDED
NO_MEANINGFUL_IMPROVEMENT
ADAPTER_UNAVAILABLE
```

Use a transparent deterministic score, not ML:

```text
net score = yield benefit - gas penalty - protocol risk penalty - liquidity penalty - concentration penalty
```

Return:

- selected candidate;
- eligible candidates;
- rejected candidates with exact codes and human-readable reasons;
- component score breakdown;
- typed action plan or `DO_NOTHING` plan;
- policy hash, snapshot hash, opportunity-set hash, and plan hash.

## 5. `plan_simulate`

Input: a typed plan produced by `strategy_createPlan`.

For an executable plan:

- confirm chain and contract allowlists;
- confirm data freshness and deadline;
- check ERC-20 allowance;
- build only known adapter calls;
- perform `eth_call`/contract simulation and gas estimation where configuration permits;
- return expected pre/postconditions;
- create a simulation hash.

For `DO_NOTHING`, return a successful no-op simulation with no transaction.

Never accept arbitrary target addresses, function selectors, or arbitrary calldata from tool input.

## 6. `execution_prepare`

Input:

- valid plan;
- matching plan/policy/snapshot hashes;
- passing simulation result/hash.

Return an unsigned EIP-1193-compatible transaction request for a browser wallet. Targets, ABI functions, arguments, chain ID, value, and calldata must be generated internally from the allowlisted adapter. Use exact/bounded approvals rather than unlimited approvals. Reject stale, mismatched, expired, unconfigured, or failed simulations.

Do not submit the transaction and never request a private key.

## 7. `execution_verify`

Input:

```json
{
  "chainId": 84532,
  "transactionHash": "0x...",
  "planHash": "0x..."
}
```

Read the live receipt and relevant post-transaction state. Return status, block number, gas used, verified postconditions, observed timestamp, and structured failure information.

## 8. `receipt_get`

Input: receipt ID or plan hash.

Return a Decision Receipt containing:

- wallet and policy summary;
- source chains, block numbers, timestamps, adapter versions, and contract addresses;
- eligible and rejected candidates;
- selected candidate and score breakdown;
- plan and hashes;
- simulation result;
- prepared transaction summary;
- verification result and transaction hash when available;
- final receipt hash.

Use an in-memory repository for the hackathon; no database is needed. Clearly state that receipts reset when the server restarts.
